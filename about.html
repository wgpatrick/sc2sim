How this works

The short version of everything the main page assumes you already know.

What this actually is

An event-driven economy simulator for StarCraft II build orders. You give it a sequence of actions (Probe, Gateway, chrono:Stalker, …) for one race, and it plays out the economy second-by-second: when each thing starts, finishes, and — for army units — when it would physically arrive at the enemy's base on a given map. There is no opponent and no combat in the simulation itself; it only answers "how fast, and with what," not "who wins the fight."

Two ways to search for a good build

Both answer the same question — "what's the fastest way to get this army to the enemy?" — but explore the space differently:

Exact searchTries every combination inside one fixed shape of build (worker count → gas → tech → units, one main production building) — a few thousand to ~30,000 variants. Guaranteed the best answer within that shape, but can't discover anything the shape doesn't allow for.
Smart searchA genetic algorithm: it mutates and cross-breeds actual build sequences directly, with no fixed shape at all. Slower per-run and not guaranteed optimal, but in practice it beats the exact search by 10–20 seconds on almost every target, because it can find orderings — an unusual early proxy, a chrono on the tech building instead of a probe — that the exact search's shape simply can't express.

"Best builds" on the main page runs the smart search against a curated set of common targets automatically, so you don't have to pick a target yourself first.

What "value" means

The Value Frontier panel ranks armies by a single number per unit: value = √(dps × (hp + shields)) — roughly, "how much sustained damage can this unit deal before it dies." An army's value is just the sum of its units' values. This is a ranking heuristic, not a combat outcome — it has no idea about range, splash damage, unit counters, concave positioning, or upgrades. A maxed-out Zealot ball with +3 weapons/armor and Charge researched is dramatically stronger in a real game than the same number of stock Zealots, and right now this tool can't tell the difference — upgrades aren't modeled anywhere yet. Treat "value" as "how big is this army," not "who would win."

What the Opponent panel is comparing

This tool has a small library of real ladder replays, parsed down to their opening build order. The Opponent panel takes one of those real, recorded build orders and runs it through the same simulator (as if it were just another build order you'd typed in), producing a value-over-time curve for what that real player actually did. It then compares that curve against whatever build you currently have loaded in the Simulator panel. If your build's cumulative value ever dips below the real opponent's at the same point in time, that's a moment a real opponent's build would have had more army than you — regardless of how fast your build looks in isolation.

Known limits, honestly:
  • No upgrades anywhere (Charge, Blink, weapon/armor tiers, etc.) — see above.
  • No branching: every build is one committed sequence from minute zero. Real play reacts to scouting; this only ever tests straight lines.
  • Combat, when modeled at all, is a simplified stat trade — no positioning, splash, or armor types.
  • Roster is incomplete for all three races (no Colossus/Templar/Carrier for Protoss, no Ghost/Thor/Battlecruiser for Terran, no Hive tier for Zerg, etc).
  • Map distances are 3 rough presets, not measured off real ladder maps.
